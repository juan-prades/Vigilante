The goal is to build a small 3D survival game called **"El Despertar del Vigía"** as a coursework deliverable for the subject **Programación Multimedia**. The grading professor is **Carlos** and is **VERY strict about code style**: if he sees code that doesn't match his style, he gets suspicious. Therefore it is CRITICAL to follow the style conventions described below to the letter.

As of this writing the project is freshly created — only the default `Assets/Scenes/SampleScene.unity` and `Assets/InputSystem_Actions.inputactions` exist. Gameplay code will be generated from scratch.

## Key facts that aren't obvious from a glance

- **Version control is Plastic SCM / Unity Version Control, not git.** This repo is not a git repository. `ignore.conf` is the Plastic ignore file (equivalent to `.gitignore`); add new ignore rules there. Note it ignores `*.sln`/`*.csproj` (regenerated by Unity) and `Assets/Plugins/PlasticSCM*`.
- **Input handling**: although the project was created with the new Input System package enabled, we are going to use the **legacy `UnityEngine.Input` API** (`Input.GetKeyDown`, `Input.GetAxis`, etc.) because that's what the professor uses in all his examples. If runtime errors appear because the legacy Input is disabled, switch the project setting `Active Input Handling` in `Edit > Project Settings > Player` to **"Both"**.
- **Every asset has a paired `.meta` file.** When moving/renaming/deleting an asset, the `.meta` must move with it; deleting one orphans GUID references in scenes/prefabs. Prefer doing asset operations inside the Unity Editor so meta files and references stay consistent.
- **API compatibility level is .NET Standard 2.1** (`apiCompatibilityLevel: 6`).
- **Render pipeline**: assume Built-In Render Pipeline. If URP or HDRP appears in `Packages/manifest.json`, warn the user.

## Editor

- Open the project with the matching Editor version (installed via Unity Hub at `C:\Program Files\Unity\Hub\Editor\6000.3.2f1`). Mismatched versions will reimport assets.
- `Library/`, `Temp/`, `Logs/`, `obj/`, and the generated `.sln`/`.csproj` are build artifacts — never edit them by hand; Unity regenerates them.

## Command line (batch mode)

Unity has no separate build/lint/test CLIs — everything runs through the Editor executable. Quote the path (it contains spaces).

```powershell
$unity = "C:\Program Files\Unity\Hub\Editor\6000.3.2f1\Editor\Unity.exe"
$proj  = "C:\Users\Trabajo\GitHub\Vigilante"
```

Headless build (after a build script exists, e.g. a static `Builder.PerformBuild` method):

```powershell
& $unity -quit -batchmode -projectPath $proj -executeMethod Builder.PerformBuild -logFile "$proj\Logs\build.log"
```

Batch-mode runs are silent on success; check the `-logFile` for outcomes. Only one Editor process can hold the project lock at a time — close the GUI Editor before running batch commands against the same project.

# Contexto del juego: "El Despertar del Vigía"

## Concepto

Juego de supervivencia en primera persona. El jugador despierta en una base abandonada de noche, las luces fallan y debe llegar a la sala de control esquivando vigilantes mecánicos que patrullan la base.

## Mecánicas

- Movimiento en primera persona (WASD + ratón + salto)
- Linterna que se enciende/apaga (teclas 4 y 5)
- Luces de la base que se controlan con teclas (1, 2, 3)
- Luces zonales que parpadean en bucle (tecla 7 para alternar)
- Enemigo patrullero que persigue al jugador si lo detecta
- Enemigo cercano que se activa al acercarte
- Torreta enemiga que dispara balas
- Música de fondo + efectos de sonido + zonas con sonido ambiente
- Cambio de cámara entre 1ª persona / 3ª persona / cenital (T/Y/U)
- Meta: llegar a un cubo final (panel de control) para ganar

# ESTILO DEL CÓDIGO (CRÍTICO - SEGUIR SIEMPRE)

## Archivo de referencia

El archivo `referencia_profesor.txt` (en la raíz del proyecto) contiene **el código exacto que el profesor publicó en clase**. Es la fuente de verdad para el estilo y para los scripts que hay que replicar tal cual.

**Si no encuentras `referencia_profesor.txt`, AVISA AL USUARIO INMEDIATAMENTE y no inventes el contenido.** El archivo debe estar en la raíz del proyecto, al mismo nivel que este `CLAUDE.md`.

## Reglas absolutas

1. **Comentarios en español** siempre. Ejemplos: `// Arrastra aquí la luz desde el inspector`, `// Empieza apagada`, `// Mirar hacia el jugador`.
2. **Nombres de clases y variables en español**: `luz`, `linterna`, `bala`, `patrullero`, `cercano`, `dispara`, `seguir`, `puntos`, `vive`, `paso`, `intensidad`, `velocidad`, `encendida`. NUNCA usar inglés en variables nuevas (excepto las que ya están en inglés en sus scripts como `MoveSpeed`, `shootRange`, etc., que mantenemos para los enemigos tal y como las escribió él).
3. **Clases en minúscula o capitalización inconsistente** (su estilo NO es PascalCase estricto): `patrullero`, `cercano`, `seguir`, `dispara`, `bala`, `ControlLuz`, `Linterna`, `LuzZonalTitileo`, `SoundManager`, `CameraSwitcher`, `ZoneTrigger`, `FirstPersonController`. Mantén estos nombres TAL CUAL.
4. **Casi todo `public`**: las variables van `public` para poder arrastrarlas desde el Inspector. Solo usar `private` cuando es un estado interno (como `fireCooldown`, `active`, `persiguiendo`).
5. **Usar `[Header("...")]`** para agrupar variables relacionadas en el Inspector. Ejemplo:
   ```csharp
   [Header("Patrulla")]
   public Transform[] puntos;
   public float patrolSpeed = 2f;
   ```
6. **NUNCA usar namespaces.**
7. **Imports al estilo del profe**: siempre que el script tenga colecciones, poner las 3 líneas:
   ```csharp
   using System.Collections;
   using System.Collections.Generic;
   using UnityEngine;
   ```
   Si es un script simple sin colecciones (como `ControlLuz`), solo `using UnityEngine;`.
8. **Debug.Log para depurar** en momentos clave: `Debug.Log("salta");`, `Debug.Log("Colisión con el jugador detectada");`. El profe los usa, nosotros también.
9. **No usar features modernas** como `record`, `init`, expression-bodied members complejos. Código procedural sencillo, estilo "primer año de C#".
10. **SoundManager con patrón Singleton** (`public static SoundManager Instance;`).
11. **Mantener simplicidad**: nada de patrones de diseño avanzados, nada de eventos C#, nada de ScriptableObjects, nada de async/await. Todo plano y directo.
12. **Llaves estilo Allman** (siempre en línea nueva, no K&R).

## Plantilla mental del estilo

Mira este ejemplo del profe y replícalo:

```csharp
using UnityEngine;

public class ControlLuz : MonoBehaviour
{
    public Light luz;              // Arrastra aquí la luz desde el inspector
    public float paso = 0.2f;      // Cantidad que sube o baja la intensidad

    void Update()
    {
        // Aumentar intensidad (tecla 1)
        if (Input.GetKeyDown(KeyCode.Alpha1))
        {
            luz.intensity += paso;
        }
    }
}
```

Fíjate en:
- Comentario al lado de cada `public` explicando qué se arrastra
- Comentario sobre cada bloque de `if` explicando qué hace
- Espacios en blanco entre bloques lógicos
- Llaves siempre en línea nueva

# Cosas que NO hacer

- NO usar `async/await`
- NO usar LINQ complejo en scripts de juego
- NO usar `[SerializeField] private` (el profe usa `public` directamente)
- NO usar tipos modernos como `record` o tuplas complejas
- NO añadir tests unitarios
- NO crear archivos de configuración fuera de `Assets/`
- NO modificar `Packages/manifest.json` salvo que sea estrictamente necesario
- NO usar el New Input System en código (usar `Input.GetKeyDown`, `Input.GetAxis`, etc.)
- NO crear assembly definitions (`.asmdef`) — añaden complejidad innecesaria para un proyecto de prácticas

# Flujo de trabajo

1. **Verificar** que `referencia_profesor.txt` existe en la raíz del proyecto. Si no, parar y avisar al usuario.
2. **Leer** `referencia_profesor.txt` completo.
3. **Crear** la estructura de carpetas dentro de `Assets/Scripts/`.
4. **Crear** todos los scripts del profe copiados tal cual del archivo de referencia.
5. **Crear** los scripts nuevos (`MetaJuego.cs`, `UIManager.cs`) siguiendo el mismo estilo.
6. **Generar** un documento `INSTRUCCIONES_UNITY.md` en la raíz con los pasos detallados para montar la escena en Unity:
   - Por cada GameObject: tipo (Cube, Capsule, Light...), tag, componentes (Rigidbody, Collider, CharacterController...), script a arrastrar, referencias `public` a asignar en el Inspector.
   - Lista numerada, paso a paso, asumiendo que el usuario no es experto en Unity.
7. **Sugerir** 3-4 sonidos gratis con URL (Pixabay, freesound.org) que el usuario debe descargar para meter en `Assets/Audio/`.

# Aviso importante sobre Input System

Si al ejecutar el juego en Unity aparece un error tipo `InvalidOperationException: You are trying to read Input using the UnityEngine.Input class, but you have switched active Input handling to Input System package in Player Settings.`:

1. Decirle al usuario que vaya a `Edit > Project Settings > Player > Other Settings > Active Input Handling`.
2. Cambiar el valor de `Input System Package (New)` a **`Both`**.
3. Unity pedirá reiniciar el Editor — aceptar.
